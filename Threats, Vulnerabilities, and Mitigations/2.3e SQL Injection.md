# Code Injection & SQL Injection

## Overview

A **code injection attack** occurs when an attacker inserts their own code into data that an application accepts as input. Instead of processing the input as plain data, the application — lacking proper validation — executes the injected code.

Code injection is one of the most common categories of application vulnerabilities. The responsibility for prevention lies with the application developer: input must be validated and sanitized before being passed to any downstream system.

### Common Injection Types

| Injection Type | Target |
|---------------|--------|
| **SQL Injection (SQLi)** | Relational databases |
| **HTML Injection** | Web page rendering |
| **XML Injection** | XML parsers and APIs |
| **Command Injection** | Operating system shell |

---

## SQL Injection (SQLi)

**SQL (Structured Query Language)** is the standard language used by applications to interact with relational databases. When a user inputs data into an application — a search field, a login form, a filter — the application typically constructs a SQL query using that input and sends it to the database.

**SQL injection** exploits applications that fail to validate this input, allowing an attacker to append or modify the SQL query being sent to the database.

### Why It's Dangerous

- Requires no additional software — exploitable directly through a browser's input fields
- No need to trick a user into clicking anything
- Can expose, modify, or destroy all data in the database
- Can bypass authentication entirely

---

## How SQL Injection Works

### Normal Query

An application accepts a username input and constructs the following query:

```sql
SELECT * FROM users WHERE name = 'Professor'
```

The database returns records for the user named "Professor" — the intended behavior.

### Injected Query

If the application does not validate input, an attacker can append additional SQL code to the input field:

```
Input: Professor' OR '1'='1
```

The resulting query sent to the database becomes:

```sql
SELECT * FROM users WHERE name = 'Professor' OR '1'='1'
```

Because `1 = 1` is always true, this condition matches **every row in the database**. The attacker receives the entire contents of the table.

> **Indicator:** Any SQL query containing a condition like `WHERE 1=1` or `OR '1'='1'` is a strong signal that SQL injection has occurred or is being attempted.

---

## What an Attacker Can Do With SQL Injection

| Action | SQL Example |
|--------|------------|
| **Read all data** | `SELECT * FROM users WHERE 1=1` |
| **Bypass authentication** | `WHERE username='admin' OR '1'='1'` |
| **Delete data** | Appending `; DROP TABLE users;` |
| **Modify data** | Injecting `UPDATE` or `INSERT` statements |
| **Crash the database** | Causing errors that make data unavailable |

Because the attacker is communicating directly with the database by circumventing the application's security layer, they effectively have the same level of access as the application itself.

---

## Step-by-Step Example

Using a deliberately vulnerable application (such as **WebGoat** at [webgoat.org](https://webgoat.org)):

**Normal login attempt:**
- Employee name: `Smith`
- Transaction auth number: `3SL99A`
- Result: Returns department data for Smith with that auth number only

**SQL injection attempt:**
- Employee name: `Smith`
- Transaction auth number: `3SL99A' OR '1'='1`

The injected query becomes:

```sql
SELECT * FROM employees WHERE name='Smith' AND auth='3SL99A' OR '1'='1'
```

Because `'1'='1'` is always true, the query returns **every record in the database** — regardless of name or auth number.

## Prevention

| Defense | Description |
|---------|-------------|
| **Parameterized queries (prepared statements)** | Separates SQL code from user input entirely — input is never interpreted as SQL |
| **Input validation** | Reject or sanitize input that contains SQL syntax characters (`'`, `;`, `--`, etc.) |
| **Stored procedures** | Encapsulates SQL logic on the database server, reducing exposure to raw query construction |
| **Least privilege** | Database accounts used by the application should have only the permissions they need — not full admin access |
| **Web Application Firewall (WAF)** | Detects and blocks common injection patterns at the network layer |
| **Error handling** | Suppress detailed database error messages — they give attackers information about the query structure |

> **The most effective defense is parameterized queries.** When implemented correctly, user input is passed as a data value — never as part of the SQL command — making injection structurally impossible regardless of what the input contains.

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Code injection** | Attacker inserts executable code into application input fields |
| **SQL injection** | Malicious SQL appended to a query, giving the attacker direct database access |
| **`OR '1'='1'`** | Classic SQLi payload — always evaluates to true, bypassing conditional filters |
| **Impact** | Full read, modify, or delete access to database contents; authentication bypass |
| **Primary defense** | Parameterized queries / prepared statements — separate code from data |
