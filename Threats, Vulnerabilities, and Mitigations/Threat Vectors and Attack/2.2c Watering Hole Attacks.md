# Watering Hole Attacks

## Overview

A **watering hole attack** is a strategy where an attacker compromises a third-party website that a target organization is known to visit — then waits for employees to visit that site and become infected. Rather than attacking the target directly, the attacker poisons a resource the target will come to on their own.

The name comes from the predatory tactic of waiting at a watering hole for prey to arrive, rather than pursuing it directly.

---

## Why Attackers Use This Approach

When an organization has strong security awareness training — employees who won't plug in unknown USB drives, won't click suspicious email links, and won't open unsolicited attachments — direct attack vectors become ineffective. A watering hole attack sidesteps these defenses entirely by targeting a trusted third party instead.

The attacker shifts the question from:
> *"How do I get into their network?"*

to:
> *"What websites do their employees visit, and can I get into one of those?"*

---

## How It Works

1. **Reconnaissance** — The attacker researches which third-party websites the target organization regularly visits (e.g., ordering sites, industry forums, vendor portals, news sites)
2. **Compromise the third-party site** — The attacker finds and exploits a vulnerability in that website, or uses phishing to gain access to its server
3. **Inject malicious code** — Malware, malicious JavaScript, or other payloads are embedded in the site
4. **Wait** — The attacker waits for employees from the target organization to visit the site as part of their normal routine
5. **Infection** — When a target visits, the malicious code executes in their browser and gains a foothold

```
[Attacker]
     |
     | (exploits vulnerability)
     ▼
[Third-Party Website] ◄── employees visit routinely
     |
     | (malicious code executes on visit)
     ▼
[Target Organization's Systems]
```

---

## Targeted vs. Broad Poisoning

Watering hole attacks can be broad (infecting all visitors) or highly targeted (infecting only specific visitors based on IP address or other identifiers).

**Targeted example:** Rather than infecting every visitor to a compromised site, the attacker injects code that only executes for IP addresses belonging to specific financial institutions. All other visitors see the normal site with no indication anything is wrong.

This makes targeted watering hole attacks especially difficult to detect — the malicious code may never be seen by the security researchers or casual visitors who would otherwise identify it.

---

## Real-World Example: Financial Regulators (January 2017)

In January 2017, attackers successfully compromised the websites of:

- The **Polish Financial Supervision Authority**
- The **National Banking and Stock Commission of Mexico**
- A **state-owned bank in Uruguay**

After gaining access, they embedded malicious JavaScript files in each site. However, the code was designed to execute **only for IP addresses associated with specific banking institutions** — all other visitors saw the unmodified site.

The full outcome of the attack was never publicly disclosed, so it is unknown whether the attackers achieved their ultimate objective. However, the incident demonstrated the sophistication of modern watering hole campaigns and the value of targeted code injection.

---

## Defense: Layered Security (Defense in Depth)

No single control can reliably stop a watering hole attack. Because the infection arrives through a trusted site via legitimate browsing traffic, it can bypass perimeter controls that would catch other threat vectors. The most effective defense is **defense in depth** — multiple overlapping security layers so that if one layer misses the threat, another catches it.

| Layer | Role in Defense |
|-------|----------------|
| **Antivirus / EDR** | Detects and blocks known malicious code before it executes |
| **Firewall** | Filters traffic; may block communication to known malicious destinations |
| **Intrusion Prevention System (IPS)** | Inspects traffic content and flags malicious payloads that the firewall allows through |
| **Browser security controls** | Restricts execution of untrusted scripts or content |
| **Network monitoring** | Detects unusual outbound traffic patterns that may indicate an infection |

> **Example:** In the 2017 Polish Financial Supervision Authority attack, users running Symantec antivirus received an alert that malicious code had been detected and blocked — preventing execution even after the compromised site was visited.

Firewalls and intrusion prevention systems are commonly deployed together for this reason: the firewall governs what traffic is allowed, while the IPS inspects the content of that allowed traffic for malicious behavior.

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Watering hole attack** | Compromising a third-party site that targets are known to visit, then waiting for them to arrive |
| **Why it works** | Bypasses direct-attack defenses by targeting trusted external resources |
| **Targeted poisoning** | Malicious code may execute only for specific IP ranges, making it invisible to most visitors |
| **Defense in depth** | No single control is sufficient — layered security (AV, firewall, IPS, monitoring) is required |
