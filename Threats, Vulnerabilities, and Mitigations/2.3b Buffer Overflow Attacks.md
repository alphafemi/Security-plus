# Buffer Overflow Attacks

## Overview

A **buffer overflow** occurs when more data is written into a section of memory than that section was designed to hold. The excess data spills — or overflows — into adjacent memory, potentially overwriting values that control how an application behaves.

Developers guard against this with **bounds checking** — validating that any input written to a memory location does not exceed its allocated size. When bounds checking is absent or flawed, a buffer overflow vulnerability exists.

---

## Why Buffer Overflows Are Difficult to Exploit

Finding a buffer overflow vulnerability is only the first step. Exploiting it reliably is significantly harder:

- **Writing unexpected data to memory often crashes the application or system** rather than producing useful results
- The attacker must find not just any overflow, but the *right* overflow — targeting a specific memory location with specific data to produce a specific, repeatable outcome
- A useful exploit must work **consistently** across multiple executions, not just by accident

What attackers look for is a buffer overflow that is:
1. **Repeatable** — produces the same result every time
2. **Controllable** — the attacker can choose what data overflows and where it lands
3. **Advantageous** — the overflow produces a result useful to the attacker (e.g., elevated privileges, code execution)

## How a Buffer Overflow Works: Step-by-Step Example

### Initial Memory State

Two variables are stored adjacent to each other in memory:

```
Variable A (8 bytes)         Variable B (2 bytes)
┌──┬──┬──┬──┬──┬──┬──┬──┐   ┌──────┐
│00│00│00│00│00│00│00│00│   │ 1979 │
└──┴──┴──┴──┴──┴──┴──┴──┘   └──────┘
```

- **Variable A** — 8 bytes, currently empty (all zeros)
- **Variable B** — 2 bytes, currently holds the value `1979`

### What Variable B Controls

In this application, Variable B determines the user's permission level:

| Value of Variable B | Permission Level |
|--------------------|-----------------|
| Below 2,000 | User / Guest rights |
| Above 24,000 | Administrator rights |

Variable B is not normally writable through the application — but the attacker has found that Variable A has no bounds checking.

---

### The Attack

The attacker writes the word `"excessive"` into Variable A. This word has **9 characters** — one more than Variable A's 8-byte capacity.

```
Input:  e  x  c  e  s  s  i  v  e
Bytes:  1  2  3  4  5  6  7  8  [9 → overflows into Variable B]
```

**Result in memory:**

```
Variable A (8 bytes)                        Variable B (2 bytes)
┌───┬───┬───┬───┬───┬───┬───┬───┐          ┌────────┐
│ e │ x │ c │ e │ s │ s │ i │ v │          │ 0x65?? │
└───┴───┴───┴───┴───┴───┴───┴───┘          └────────┘
                                    ↑
                          The letter 'e' (hex: 0x65)
                          overflows into Variable B
```

The ninth character — `e` — has a hexadecimal value of `0x65`. When written into the first byte of Variable B, it changes Variable B's value to **25,856**.

### Outcome

| | Before Attack | After Attack |
|--|--------------|-------------|
| **Variable B value** | 1,979 | 25,856 |
| **Permission level** | User / Guest | **Administrator** |

The attacker has gained administrative rights **without ever providing administrator credentials** — by writing one extra byte into an adjacent variable.

---

## Why This Is Dangerous

Buffer overflows can be used to:

- **Escalate privileges** — as shown above, overwriting permission-related variables
- **Execute arbitrary code** — overwriting a return address in memory to redirect execution to attacker-controlled code
- **Crash systems** — even failed exploits can cause denial of service
- **Bypass authentication** — overwriting variables that store authentication state

---

## Prevention

| Defense | Description |
|---------|-------------|
| **Bounds checking** | Validate that all input fits within its allocated buffer before writing |
| **Memory-safe languages** | Languages like Rust perform automatic bounds checking; C and C++ do not by default |
| **Address Space Layout Randomization (ASLR)** | Randomizes memory addresses, making it harder to predict where to overflow data |
| **Data Execution Prevention (DEP)** | Marks memory regions as non-executable, preventing injected code from running |
| **Stack canaries** | Places a known value between buffers and control data; if the canary is overwritten, the program detects the overflow and terminates |

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Buffer overflow** | Writing more data than a memory buffer can hold, causing data to spill into adjacent memory |
| **Bounds checking** | Developer-side validation that prevents writing beyond an allocated buffer |
| **Repeatable exploit** | Attackers need a consistent, controllable overflow — not just any crash |
| **Privilege escalation via overflow** | Overwriting adjacent variables can grant elevated permissions without credentials |
| **Mitigations** | Bounds checking, ASLR, DEP, stack canaries, and memory-safe languages all reduce risk |
