# Cloud Security Vulnerabilities

## Overview

Cloud adoption has grown rapidly — today, most organizations run at least one application in a public cloud. With those applications comes sensitive data, and securing it requires the same rigor as on-premises infrastructure, plus awareness of cloud-specific risks.

Despite this, security practices in the cloud consistently lag behind:

| Statistic | Detail |
|-----------|--------|
| **76%** of organizations | Do not use multifactor authentication for their cloud management console |
| **63%** of cloud code | Is currently unpatched — often with CVSS scores ≥ 7 out of 10 |

> **CVSS (Common Vulnerability Scoring System)** rates vulnerability severity on a 0–10 scale. A score of 7 or higher is considered high to critical.

---

## The Public Cloud Attack Surface

When an application is deployed in a public cloud, it is reachable by anyone in the world — that is, by design, the entire point. But this also means:

- **Any attacker** in the world can attempt to connect to it
- **Any attacker** can attempt to disable it via Denial of Service (DoS) or Distributed Denial of Service (DDoS)
- **Any misconfiguration** in authentication, access control, or network settings is potentially exposed globally

The consequence of a misconfiguration or unpatched vulnerability is therefore much greater in a public cloud than in an internal system protected by a network perimeter.

---

## Common Cloud Vulnerabilities

### 1. Weak or Missing Authentication

Without strong authentication — particularly **multifactor authentication (MFA)** — cloud management consoles and applications are vulnerable to credential-based attacks. A compromised admin account can give an attacker full control of an organization's entire cloud environment.

### 2. Unpatched Systems & Applications

Unpatched operating systems and applications in the cloud carry the same risks as on-premises — but the exposure is global. Recent high-profile examples include:

| Vulnerability | Impact |
|--------------|--------|
| **Log4j** | Extremely easy to exploit; grants full control of the affected system with minimal technical knowledge |
| **Spring Cloud Function** | Similar severity; remote code execution with broad access to the underlying system |

Both vulnerabilities allowed attackers to pivot from the application layer into the underlying OS and potentially into other systems within the same cloud environment.

### 3. Misconfiguration

Cloud misconfigurations are one of the leading causes of data breaches. Common examples include:

- **Directory traversal** — web servers that allow users to navigate folder structures and access files outside the intended web root
- **Overly permissive access controls** — storage buckets or databases left publicly readable
- **Missing authentication** — APIs or management interfaces accessible without credentials

### 4. Remote Code Execution (RCE)

When a system is unpatched or an application accepts unsanitized input, an attacker may be able to execute arbitrary code on the cloud system. Because cloud systems often have broad internal network access, RCE can quickly escalate into a much larger breach.

### 5. Out-of-Bounds Write

An **out-of-bounds write** occurs when an attacker writes data to a memory location they are not authorized to access. This can:

- Enable remote code execution
- Crash the application, causing a denial of service for legitimate users

### 6. Cross-Site Scripting (XSS)

Applications that lack input validation are vulnerable to XSS attacks. In a cloud context, this can allow attackers to steal session tokens, hijack authenticated user sessions, or use the trusted site as a delivery mechanism for malware.

### 7. Injection Attacks (SQL Injection)

When application data lives in a cloud database, SQL injection and other code injection attacks can give an attacker direct access to that data — potentially exposing all records in the database.

## Cloud Vulnerability Attack Chain

Many cloud breaches follow a common escalation pattern:

```
[Initial Access]
Exploit unpatched app (Log4j, XSS, SQLi, etc.)
          |
          ▼
[Application Compromise]
Gain control of the cloud application
          |
          ▼
[OS-Level Access]
Pivot from application to underlying OS via RCE
          |
          ▼
[Lateral Movement]
Move from compromised system to other systems
within the same cloud environment
          |
          ▼
[Data Exfiltration or Disruption]
Access, steal, or destroy cloud-hosted data
```

## Mitigation

| Vulnerability | Mitigation |
|--------------|-----------|
| **Weak authentication** | Enforce MFA on all cloud consoles and administrative access |
| **Unpatched systems** | Establish a regular patching cadence for both OS and application layers |
| **Misconfiguration** | Use cloud security posture management (CSPM) tools; audit access controls regularly |
| **Directory traversal** | Disable directory browsing on web servers; validate and restrict file path inputs |
| **RCE / out-of-bounds** | Patch promptly; use runtime application self-protection (RASP) and WAFs |
| **XSS** | Validate and encode all user inputs; implement Content Security Policy |
| **SQL injection** | Use parameterized queries; apply least-privilege database permissions |

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Public cloud exposure** | Cloud applications are reachable by anyone — misconfigurations and vulnerabilities are globally exploitable |
| **MFA gap** | 76% of organizations lack MFA on cloud consoles — a critical baseline control |
| **Unpatched code** | 63% of cloud code is unpatched, often at high or critical severity |
| **Log4j / Spring Cloud** | Easy-to-exploit vulnerabilities with full system access as the reward |
| **Directory traversal** | Web server misconfiguration that allows unauthorized access to file system directories |
| **Attack chain** | Application exploit → OS access → lateral movement → data breach |
| **Defense** | MFA, regular patching, input validation, access control audits, and network segmentation |
