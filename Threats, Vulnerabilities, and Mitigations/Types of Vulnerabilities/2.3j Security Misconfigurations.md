# Common Security Misconfigurations

## Overview

Many of the most damaging breaches in history weren't the result of sophisticated zero-day exploits — they were caused by basic security misconfigurations. Open cloud storage, weak admin passwords, unencrypted protocols, and unchecked firewall rules are all preventable problems that attackers actively and routinely look for.

---

## Open Cloud Storage

Attackers use automated reconnaissance to scan for publicly accessible cloud storage — S3 buckets, blob storage, and other repositories left open without access controls. They know that organizations frequently misconfigure storage and that eventually they will find exposed data.

### Real-World Example: Verizon (June 2017)

A third-party vendor left an **Amazon S3 data repository open with no password or access controls**, exposing **14 million Verizon customer records** to the public internet.

- The data was discovered by a security researcher, not an attacker
- The repository was closed and no confirmed unauthorized downloads occurred
- Had an attacker found it first, 14 million records would have been freely available for download and distribution

> Unintentional public exposure of data is common enough that attackers have built automated tools specifically to find it.

**Prevention:**
- Audit cloud storage permissions regularly
- Default to private access — require explicit action to make any storage public
- Use cloud security posture management (CSPM) tools to flag open buckets automatically

## Unsecured Admin Accounts

Every operating system has a superuser account — **root** in Linux, **Administrator** in Windows. These accounts have unrestricted access to the entire system, making them a prime target.

### Common Mistakes

| Mistake | Risk |
|---------|------|
| Weak or guessable password (`123456`, `admin`, `football`) | Trivially brute-forced |
| Leaving the root/admin account enabled for direct login | Direct attack surface for credential attacks |
| Too many accounts with admin privileges | Larger attack surface; more opportunities for compromise |

### Best Practices

- **Disable direct root/admin login** — require users to first authenticate with their standard account, then elevate using `sudo` (Linux) or "Run as Administrator" (Windows)
- **Use strong, unique passwords** on any account that cannot be disabled
- **Limit the number of admin-level accounts** — the fewer accounts with elevated privileges, the smaller the attack surface
- **Audit admin accounts regularly** — remove access that is no longer needed

## Unencrypted Protocols

Encryption protects data in transit — but only if the protocols in use actually support it. Several common legacy protocols transmit all data in **plaintext**, meaning anyone who can capture network traffic can read it directly.

### Insecure vs. Secure Protocol Alternatives

| Insecure Protocol | Transmits | Secure Alternative |
|------------------|-----------|-------------------|
| **HTTP** | Web traffic | **HTTPS** |
| **Telnet** | Remote terminal sessions | **SSH** |
| **FTP** | File transfers | **SFTP / FTPS** |
| **SMTP** (unencrypted) | Email sending | **SMTP with TLS** |
| **IMAP / POP3** (unencrypted) | Email retrieval | **IMAPS / POP3S** |

### The "Wall of Sheep"

At **DEF CON 22**, a major security conference, organizers monitored all wireless traffic on the network. Any data sent over insecure protocols (HTTP, IMAP, POP3) was extracted and displayed publicly — including:

- Email addresses
- Partial passwords
- IP addresses
- Cookie data and session details

This public display — called the **"wall of sheep"** — illustrated in real time how much information is visible when unencrypted protocols are used. If attendees at a security conference were sending data in the clear, the problem is widespread.

**Prevention:** Perform a packet capture on your network using a tool like Wireshark. If you can read the contents of your own traffic, so can an attacker with network access.

## Default Credentials

Every network device ships with a default username and password. Many devices prompt users to change these on first login — but many do not, and many users skip the step when they do.

### The Mirai Botnet

The **Mirai botnet** specifically targets IoT devices using default credentials. It maintains a list of over **60 default username/password combinations** for common devices including:

- IP cameras
- Routers and switches
- Doorbells and smart locks
- Garage door openers
- Any other network-connected IoT device

When Mirai finds a device using default credentials, it compromises it and adds it to the botnet. Mirai's source code has been publicly released, meaning both researchers and attackers can run it freely.

**Prevention:**
- Change default credentials on every device immediately upon deployment
- Conduct periodic network scans to identify devices still using default configurations
- Disable remote access on devices that don't require it

## Unnecessary Open Ports

Every service running on a server opens a port that is accessible from outside. Each open port is a potential entry point. The more services running, the larger the attack surface.

### Firewall Rule Complexity

Firewalls manage port access — but large, complex rule sets introduce their own risks:

- Rules accumulate over time and are rarely cleaned up
- An administrator may accidentally allow access to a port or IP that should be restricted
- Open ports that go unnoticed are eventually discovered by attackers through port scanning

**Prevention:**

| Action | Description |
|--------|-------------|
| **Principle of least access** | Only open ports that are actively required; close everything else |
| **Regular firewall audits** | Periodically review the entire rule base for unnecessary or misconfigured rules |
| **Port scanning your own network** | Run scans from an external perspective to see what attackers would see |
| **Remove unused services** | If a service is not needed, disable it entirely rather than relying on a firewall rule |

---

## Summary

| Misconfiguration | Key Takeaway |
|-----------------|-------------|
| **Open cloud storage** | Automated tools scan for exposed buckets — default to private and audit regularly |
| **Weak admin credentials** | Root/admin accounts with simple passwords are trivially brute-forced |
| **Direct root login enabled** | Disable it; require elevation from a standard account instead |
| **Unencrypted protocols** | Telnet, FTP, HTTP, and others transmit data in plaintext — switch to secure alternatives |
| **Default credentials** | Botnets like Mirai actively scan for devices using factory defaults — change them immediately |
| **Unnecessary open ports** | Every open port is an attack surface — audit firewalls regularly and close unused ports |
