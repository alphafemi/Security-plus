# Race Conditions & TOCTOU Attacks

## Overview

A **race condition** occurs when two or more events happen nearly simultaneously within an application, and the application fails to account for the possibility that they may be executing at the same time. The result is an unexpected — and potentially exploitable — outcome.

Race conditions are a known challenge in application development. Developers typically test for them during the build process, but complex applications can have many possible event combinations, and edge cases are sometimes missed.

---

## TOCTOU — Time-Of-Check to Time-Of-Use

**TOCTOU** is one of the most common race condition attack types. It exploits the gap between two moments:

1. **Time of Check** — the application reads or checks a value from the system
2. **Time of Use** — the application acts on that value

If another process changes the value **between** those two moments — and the application doesn't account for this — the action is taken based on stale data. An attacker who can control what happens in that gap can manipulate the outcome.

```
[Application checks value] ──► [Gap: another process changes the value] ──► [Application acts on old value]
         ↑                                        ↑
    Time of Check                          Race condition occurs here
```

## Practical Example: Banking Transfer Race Condition

### Setup

Two users are transferring money between two accounts:

| Account | Starting Balance |
|---------|-----------------|
| Account A | $100 |
| Account B | $100 |

**Application behavior:**
- Deposits are reflected **immediately** in the ledger
- Withdrawals are **not** immediately reflected for all users

### Step-by-Step Walkthrough

| Step | Action | Account A | Account B |
|------|--------|-----------|-----------|
| Start | Initial state | $100 | $100 |
| User 1 checks balances | Sees A=$100, B=$100 | $100 | $100 |
| User 2 checks balances | Sees A=$100, B=$100 | $100 | $100 |
| User 1 deposits $50 → Account B | Reflected immediately | $100 | $150 |
| User 2 deposits $50 → Account B | Reflected immediately | $100 | $200 |
| User 1 withdraws $50 from Account A | User 1 sees: A=$50, B=$200 | $50 | $200 |
| User 2 withdraws $50 from Account A | Not yet reflected for User 2 — User 2 still sees A=$100 | **$50** (actual) | $200 |

### Result

From User 2's perspective, Account A ends at **$50** — but it should be **$0**. The withdrawal by User 1 was not reflected in time, and User 2's transaction was processed against a stale balance.

This is the race condition: the application's failure to immediately synchronize withdrawal data allowed both users to act on the same starting value, producing an incorrect final state.

## Real-World Examples

### Mars Rover Spirit (2004)

A race condition in the Mars rover Spirit's software caused it to enter a **reboot loop**:

1. The rover detected a file system error
2. Its safety programming triggered an automatic reboot to correct errors
3. On reboot, it encountered the same file system error again
4. The cycle repeated indefinitely

Engineers resolved the issue by transmitting updated code to the rover to bypass the error condition — a significant challenge given the communication delay between Earth and Mars.

### Tesla Model 3 — Pwn2Own Vancouver (2023)

Researchers exploited a TOCTOU vulnerability in the Tesla Model 3's **infotainment system** via Bluetooth:

- The attack took advantage of a race condition in the infotainment system's access controls
- Researchers successfully escalated privileges to **root user** on the system
- The exploit earned the researchers a **$100,000 prize** — and they kept the Tesla

This example highlights that race conditions are not limited to traditional software — they can affect embedded systems, vehicles, and any platform where concurrent processes share state.

## Why Race Conditions Are Hard to Prevent

- **Timing-dependent** — the vulnerability only triggers when events occur in a specific order within a narrow window, making it difficult to reproduce consistently during testing
- **Concurrency is complex** — modern applications intentionally run many processes simultaneously, creating many potential interaction points
- **Edge cases multiply** — the more events an application handles, the more combinations of simultaneous events exist to test

---

## Prevention

| Defense | Description |
|---------|-------------|
| **Atomic operations** | Ensure that check-and-use steps happen as a single, uninterruptible operation |
| **Locks / mutexes** | Prevent multiple processes from accessing shared resources simultaneously |
| **Immediate state updates** | Reflect all changes (not just deposits) in real time so all processes see current data |
| **Input validation at use time** | Re-validate data immediately before use, not just at the time of initial check |
| **Concurrency testing** | Specifically test for simultaneous event interactions during development and QA |

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Race condition** | Two simultaneous events produce an unexpected outcome the application didn't account for |
| **TOCTOU** | The value used by an application has changed between when it was checked and when it was acted on |
| **Banking example** | Inconsistent update timing allows two users to act on the same stale balance, producing incorrect results |
| **Mars rover Spirit** | A race condition between error detection and rebooting created an infinite reboot loop |
| **Tesla Pwn2Own 2023** | A TOCTOU vulnerability in the infotainment system allowed privilege escalation to root via Bluetooth |
| **Prevention** | Atomic operations, locks, real-time state updates, and targeted concurrency testing |
