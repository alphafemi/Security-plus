# Memory Injection

## Overview

All software on a computer — including malware — must be loaded into **memory** to execute. The CPU processes nothing that hasn't first been loaded from disk into RAM. This means that for malware to operate, it must find a way to get itself into memory and run.

Once in memory, malware has two primary options:

| Approach | Description |
|----------|-------------|
| **Run as its own process** | Malware operates as a standalone, visible process in memory |
| **Inject into an existing process** | Malware embeds itself within a legitimate running process |

Injection into an existing process is far more dangerous — and far more common in sophisticated attacks.

---

## Why Process Injection?

When malware runs as its own process, security tools can potentially identify it as an unknown or suspicious process. Injecting into a legitimate process addresses both detection and privilege problems simultaneously:

- **Evades detection** — anti-malware scanning for malicious processes may overlook code running inside a trusted application
- **Inherits permissions** — the injected code runs with the same rights and privileges as the host process, providing instant privilege escalation without any additional exploit

```
[Legitimate Process in Memory]
┌─────────────────────────────┐
│  Start Address              │
│                             │
│  [Normal application code]  │
│                             │
│  [Injected malware code] ◄──┼── attacker injects here
│                             │
│  End Address                │
└─────────────────────────────┘
```

If the host process runs with administrator or system-level privileges, the malware immediately inherits those same privileges — without needing to exploit a separate escalation vulnerability.

---

## DLL Injection

**DLL injection** is one of the most common forms of process injection. A **DLL (Dynamic-Link Library)** is a type of executable file in Windows that contains reusable code and resources. Multiple applications can reference and load the same DLL simultaneously, making them a natural target for abuse.

### How DLL Injection Works

1. **The attacker places a malicious DLL** on storage accessible to the target system
2. **A reference to the malicious DLL's path** is inserted into the target process — either through a configuration file, registry entry, or by manipulating the process directly
3. **The target process executes normally** until it reaches the point where it needs to load a DLL
4. **The process loads the malicious DLL from disk into memory** — treating it as a legitimate library
5. **The malware is now running** inside the context of the trusted process

```
[Target Process Executing]
        |
        | (reaches DLL reference)
        ▼
[Looks up DLL path on disk]
        |
        | (finds attacker's malicious DLL)
        ▼
[Loads malicious DLL into memory]
        |
        ▼
[Malware runs with process's rights and permissions]
```

### Why DLL Injection Is Effective

- The malicious DLL is loaded by a **legitimate, trusted process** — making it appear normal to many security tools
- The attacker does not need to exploit a privilege escalation vulnerability separately; the injection itself provides elevated rights if the host process has them
- The technique is well-understood and has many documented implementations, making it accessible to a wide range of threat actors

---

## Detection Challenges

Memory injection attacks are difficult to detect because:

- **The malicious code runs inside a trusted process** — process-level monitoring alone is insufficient
- **No standalone malicious executable** may appear on disk after the DLL is loaded
- **The DLL itself may look benign** until its behavior is analyzed at runtime

Effective defenses include behavioral monitoring (watching what processes *do*, not just what they *are*), memory scanning, and integrity checks on loaded modules.

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Memory execution** | All code must be loaded into memory to run — including malware |
| **Process injection** | Malware embeds itself in a legitimate running process to evade detection and inherit privileges |
| **Privilege escalation via injection** | Injected code automatically gains the same rights as the host process |
| **DLL (Dynamic-Link Library)** | A shared executable file that multiple processes can load; a common injection target |
| **DLL injection** | Attacker places a malicious DLL on disk and inserts a reference to it inside a target process, which then loads it automatically |
