# Cross-Site Scripting (XSS)

## Overview

**Cross-Site Scripting (XSS)** is one of the most common vulnerabilities in web applications. It occurs when an attacker is able to inject malicious scripts — typically JavaScript — into input fields or links associated with a trusted website. When the victim's browser loads that content, it executes the malicious script in the context of the trusted site.

> **Why XSS, not CSS?** The abbreviation CSS was already taken by Cascading Style Sheets, so the security community adopted **XSS** to avoid confusion.

XSS exploits the trust a browser extends to websites it considers legitimate. Because the script runs in the context of a trusted site, the browser does not block it — making XSS a powerful mechanism for stealing session data, credentials, and other sensitive information.

---

## How XSS Works (High Level)

```
[Attacker]
    |
    | (sends malicious link)
    ▼
[Victim clicks link]
    |
    | (connects to trusted website + malicious script runs)
    ▼
[Trusted Website]
    |
    | (script executes in victim's browser)
    ▼
[Victim's browser sends session data, cookies, credentials → Attacker]
```

The victim sees the trusted website normally. The malicious script runs silently in the background, exfiltrating private data directly to the attacker.

---

## XSS Attack Types

### 1. Non-Persistent (Reflected) XSS

The malicious script is embedded in a link. The attacker sends this link to the victim — via email, text, or any other channel. When the victim clicks it, the script is "reflected" off the trusted website and executed in their browser.

**Key characteristics:**
- The script is not stored on the server
- Only affects users who click the specific malicious link
- The attacker must deliver the link to each target individually

**Common target:** Input fields such as search boxes that display user-entered content back on the page without sanitizing it

### 2. Persistent (Stored) XSS

The attacker posts malicious content — including a JavaScript payload — directly to a website or platform (e.g., a social media post, a comment section, a forum). The script is stored on the server and executes in the browser of **every user who views that content**.

**Key characteristics:**
- The script is stored on the target server
- Automatically affects every visitor who loads the page
- Can spread virally if the malicious post can be shared or re-posted by victims

```
[Attacker posts malicious script to social media]
          |
          ▼
[Script stored on platform server]
          |
          ▼
[User A views post → script executes → data sent to attacker]
[User B views post → script executes → data sent to attacker]
[User C shares post → more users exposed → ...]
```

---

## What Attackers Can Steal

- **Session IDs / tokens** — allows the attacker to impersonate the victim on that site
- **Cookie data** — authentication cookies that keep a user logged in
- **Credentials** — if the script captures keystrokes on a login form
- **Personal or financial data** — anything visible or enterable on the page

---

## Example: Credit Card Field Injection

A shopping site fails to validate its credit card input field. An attacker embeds a JavaScript snippet alongside the credit card number:

```javascript
<script>alert(document.cookie)</script>
```

When the form is submitted, the script executes and exposes the session cookie. In a real attack, instead of displaying an alert, the script would silently send that cookie to an attacker-controlled server — giving the attacker the same session access as the victim.

---

## Real-World Example: Subaru XSS Vulnerability (2017)

Security researcher Aaron Guzman discovered a combination of vulnerabilities on Subaru's web portal:

| Vulnerability | Detail |
|--------------|--------|
| **XSS on Subaru site** | Allowed an attacker to steal a user's authentication token via a malicious link |
| **Non-expiring tokens** | Once stolen, the token granted access indefinitely — no re-authentication required |
| **Overprivileged tokens** | The token allowed full vehicle management — unlock, locate, start, and more |
| **Account takeover** | Adding an attacker's email to a victim's account granted the same token-level access to their vehicle |

An attacker who combined these flaws could:
1. Send a victim a malicious link
2. Steal their Subaru token via XSS
3. Use that token — forever — to control the victim's vehicle

Guzman reported the findings to Subaru through responsible disclosure, and the vulnerabilities were patched.

## Prevention

### For Users

| Action | Benefit |
|--------|---------|
| **Do not click links from untrusted sources** | Prevents reflected XSS delivery via email or message |
| **Type URLs directly into the browser** | Bypasses malicious links entirely |
| **Keep browsers and applications updated** | Patches close XSS vulnerabilities as they are discovered |
| **Consider limiting JavaScript via browser extensions** | Reduces attack surface, though may break some sites |

### For Developers

| Practice | Description |
|----------|-------------|
| **Input validation** | Reject input containing script tags or other HTML/JS syntax |
| **Output encoding** | Encode user-supplied data before rendering it in the page so it displays as text, not executable code |
| **Content Security Policy (CSP)** | HTTP header that restricts which scripts the browser is allowed to execute |
| **Token expiration** | Authentication tokens should expire — indefinite tokens dramatically increase the damage of a theft |
| **HttpOnly cookies** | Prevents JavaScript from accessing session cookies, limiting what XSS can exfiltrate |

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **XSS** | Injecting malicious scripts into trusted websites to execute in victims' browsers |
| **Reflected XSS** | Script embedded in a link; only affects users who click that specific link |
| **Stored XSS** | Script saved on the server; executes for every user who views the page |
| **Session hijacking** | Stolen session tokens allow an attacker to impersonate the victim |
| **Subaru (2017)** | XSS + non-expiring tokens + overprivileged tokens = full remote vehicle access |
| **Key developer defense** | Validate and encode all user input; implement Content Security Policy |
