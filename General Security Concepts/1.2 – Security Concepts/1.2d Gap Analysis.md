This  provides a clear and structured introduction to **gap analysis** in the context of IT security. It explains what a gap analysis is, why it is important, and how to perform one using established baselines such as NIST SP 800-171 or ISO/IEC 27001. It also covers the process of evaluating people, policies, and systems, and how to present findings in a final report.



---

### Gap Analysis for IT Security Cheat Sheet

This guide covers the purpose, process, and deliverables of a security gap analysis.

---

#### What is a Gap Analysis?

| Aspect | Description |
| :--- | :--- |
| **Definition** | A study that compares the **current state** of security with a **desired future state** (baseline or standard). |
| **Purpose** | Identify what is missing (the “gap”) and develop a plan to close that gap. |
| **Output** | A formal report documenting findings, risks, and a roadmap to achieve compliance or improved security posture. |

**Simple formula:** `Gap = Desired State – Current State`

---

#### Why Perform a Gap Analysis?

| Reason | Explanation |
| :--- | :--- |
| **Compliance** | Meet regulatory or industry standards (e.g., GDPR, PCI DSS, NIST, ISO 27001). |
| **Risk reduction** | Identify and prioritise security weaknesses before they are exploited. |
| **Resource planning** | Justify budget and staffing for security improvements. |
| **Continuous improvement** | Establish a baseline and measure progress over time. |

---

#### Common Security Baselines (Desired States)

A baseline is a set of security controls or requirements used as the target for comparison.

| Standard | Full Name / Document | Typical Use |
| :--- | :--- | :--- |
| **NIST SP 800‑171 Rev 2** | Protecting Controlled Unclassified Information (CUI) in Nonfederal Systems and Organizations | US government contractors; defence industrial base. |
| **ISO/IEC 27001** | Information Security Management Systems (ISMS) | International standard for security management; broad applicability. |
| **PCI DSS** | Payment Card Industry Data Security Standard | Organisations handling credit card data. |
| **CIS Benchmarks** | Center for Internet Security Benchmarks | Platform‑specific hardening guides. |
| **Internal standards** | Organisation‑specific policies and requirements | Custom baselines tailored to unique business needs. |

**Note:** Gap analysis can also be performed against **internal policies** rather than an external standard.

---

#### Areas of Analysis

A thorough gap analysis examines three broad areas:

| Area | What to Evaluate |
| :--- | :--- |
| **People** | Formal security experience, training history, knowledge of security policies and procedures. |
| **Processes** | Existence and adherence to security policies; formal documentation; change management; incident response. |
| **Technology** | Existing systems, configurations, vulnerabilities, security controls (firewalls, IDS/IPS, endpoint protection, logging). |

**Example (from NIST 800‑171, Access Control family):**

A broad requirement like “limit system access to authorised users” breaks down into smaller controls:

| Sub‑control | What to Assess |
| :--- | :--- |
| User registration and de‑registration | How are accounts created and removed? |
| User access provisioning | Is provisioning approved and documented? |
| Management of privileged access rights | Who has admin rights? Is it reviewed? |
| Review of user access rights | How often are permissions audited? |

**Key Point:** Start with high‑level categories, then decompose into individual, measurable security controls.

---

#### The Gap Analysis Process

| Phase | Activities |
| :--- | :--- |
| **1. Select baseline** | Choose one or more standards (NIST, ISO, PCI DSS, internal). |
| **2. Gather data** | Interviews, policy reviews, vulnerability scans, configuration audits, log analysis. |
| **3. Compare (current vs. desired)** | For each control: <br> – **Compliant / Met** <br> – **Partially met** <br> – **Not met** |
| **4. Identify gaps** | Document where current state falls short of baseline. |
| **5. Prioritise** | Rank gaps by risk (high/medium/low). |
| **6. Develop remediation plan** | Actions, timeline, resources, budget, change control. |
| **7. Produce final report** | Summary, detailed findings, recommendations, roadmap. |

**Timeline:** A full security gap analysis can take weeks, months, or even years depending on the size and complexity of the organisation.

---

#### Presenting Findings (The Gap Analysis Report)

The final report typically includes both a high‑level dashboard and detailed supporting evidence.

**Example Dashboard (Red / Yellow / Green):**

| Remote Site | Access Control | Configuration Mgmt | Incident Response | Overall |
| :--- | :--- | :--- | :--- | :--- |
| Site A | 🟢 (Good) | 🟡 (Partial) | 🔴 (Needs work) | 🟡 |
| Site B | 🔴 | 🟢 | 🟢 | 🟡 |
| Site C | 🟡 | 🟡 | 🔴 | 🔴 |
| … | … | … | … | … |

**Colour Key:**

| Colour | Meaning | Priority |
| :--- | :--- | :--- |
| 🟢 Green | Meets or nearly meets baseline | Low priority (maintain) |
| 🟡 Yellow | Partially meets baseline; some gaps | Medium priority (plan improvements) |
| 🔴 Red | Significant gaps; far from baseline | **High priority** (immediate action) |

**Recommendation:** Start remediation at sites/controls marked **red** (biggest impact), then **yellow**, then **green**.

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **Gap analysis compares current vs. desired security state** | It identifies what is missing and how to get there. |
| **Choose a baseline** (NIST, ISO, PCI DSS, internal) | The baseline defines the “desired state.” |
| **Analyse people, processes, and technology** | All three areas must be evaluated. |
| **Break broad requirements into measurable controls** | High‑level categories are not enough; assess individual sub‑controls. |
| **Use colour‑coded dashboards** (Red/Yellow/Green) | Quick visualisation of priorities for leadership. |
| **Remediate red first** | Focus resources where the gap is largest and risk is highest. |
| **Gap analysis is an ongoing process** | As systems and threats change, repeat the analysis periodically. |
