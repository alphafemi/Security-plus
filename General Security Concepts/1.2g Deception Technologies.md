## Overview

As IT security professionals work to prevent attackers from gaining access to systems, they can also use their security knowledge to create deception and disruption against attackers.

One way to provide deception is by using a honeypot—a way to attract attackers to a system and keep them involved in these systems so that you can see what type of security techniques they're trying to use.

In most cases, the attacker is actually an automated process. The goal is to see what type of automation is being used and what type of systems they are trying to attack.

---

## Honeypots

### What is a Honeypot?

A honeypot is a virtual world that effectively attracts automated systems or attackers. Attackers spend their time trying to identify or attack systems which in reality are not part of production processes.

You can build your own honeypot using a number of commercial and open-source software packages.

### The Challenge

There is a race between creating virtual worlds (which in most cases are not production systems) and attackers that are trying to discern whether these systems are actual systems or if they are trapped inside of a honeypot.

As attackers get better with identifying a honeypot, you increase the complexity and intelligence of your honeypots to make them more realistic.

## Honeynets

### What is a Honeynet?

It is very common to combine a number of virtualized honeypots into much larger infrastructures called honeynets.

These honeynets may consist of:
- Workstations
- Servers
- Routers
- Firewalls
- Anything else to make the entire infrastructure look more real to the attacker

Once you combine all of these smaller honeypots into one much larger honeynet, you've created a much more believable environment that will hopefully keep attackers very busy.

### Learning More

To learn more about the techniques and technologies used to create these honeypots and honeynets, you can visit **projecthoneypot.org**.

## Honeyfiles

### What are Honeyfiles?

You can go down to the file level and create honeyfiles. These are files that have fake information, or they may be files that appear to be very important or contain sensitive information.

### Purpose

In your normal production network, no one should be accessing these honeyfiles. So if someone does gain access to the file and opens or views the information, you may want to have alerts or alarms sent back to a management station so that you know someone is poking around in the honeyfiles who probably should not be there.

### Example

You might have a honeyfile called **passwords.txt**, which does not actually contain the passwords to your systems. But the attacker doesn't know that. They may find this to be a very attractive file and spend a lot of time going through the information contained within that honeyfile.

## Honeytokens

### What are Honeytokens?

Another type of data that might help you identify issues with data that's being released into the public is a honeytoken. Honeytokens are traceable data that you add to your honeynet. If that information is copied and distributed, you know exactly where it came from.

### Purpose

Honeytokens allow you to:
- Track if data is stolen and distributed
- Identify where data was accessed
- Monitor for copied or leaked information

### Examples

**API Credentials**

You might put API credentials out on a public cloud share to see who may come by and grab those credentials. These API credentials are not actual usable API credentials—you've simply made them up and put them into a file that is then accessed by the attacker.

**Email Addresses**

You might have a file that contains a number of fake email addresses. Because these email addresses are not used by anyone, you can constantly monitor for those addresses to appear somewhere else on the internet. If they do, you can see exactly who posted it, which might give you information about who may be attacking your network.

**Other Data Types**

Honeytokens can be any type of data that you might falsify and put into an area for an attacker to find. This could include:
- Database records
- Browser cookies
- Pixels on a web page
- Anything else that you could track if it happens to be posted somewhere else on the internet
