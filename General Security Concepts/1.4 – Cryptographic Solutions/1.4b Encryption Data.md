# Encryption Concepts

## Overview

Encryption is used to protect data in three primary states:

- **At rest** — data stored on drives and devices
- **In a database** — structured data requiring selective protection
- **In transit** — data moving across a network

---

## Encrypting Data at Rest

Data at rest refers to any data stored on a physical or virtual storage device, including individual files and entire volumes.

### Full Disk / Volume Encryption

Encrypts everything on a storage device, making all data unreadable without authentication.

| Operating System | Built-in Tool |
|-----------------|---------------|
| Windows | **BitLocker** |
| macOS | **FileVault** |
| Linux / Other | Various OS-level tools |

### File-Level Encryption

Encrypts individual files or folders rather than an entire volume.

- **Windows (EFS)** — Encrypting File System, built into NTFS. Enabled via: `File/Folder Properties → Advanced Attributes → Encrypt contents to secure data`
- **Third-party utilities** — Available for Windows, macOS, and Linux

---

## Encrypting Database Data

Databases often contain a mix of sensitive and non-sensitive data, so different encryption strategies exist to balance security with performance.

### Full Database Encryption

Uses a **symmetric key** to encrypt everything in the database.

- All data is unreadable without decryption
- Every read/write operation requires encryption or decryption
- Higher overhead, especially for large or frequently queried databases

### Column-Level Encryption

Encrypts only specific columns containing sensitive data (e.g., Social Security numbers), leaving other columns in plain text.

**Example — Employee Database:**

| Employee ID | First Name | Last Name | Social Security Number |
|-------------|------------|-----------|------------------------|
| 1001 | Jane | Smith | `[ENCRYPTED]` |
| 1002 | John | Doe | `[ENCRYPTED]` |

**Benefits:**
- Non-sensitive columns (ID, name) remain searchable without decryption overhead
- Sensitive columns are protected and only decrypted when explicitly accessed

> **Tip:** Column-level encryption is a practical middle ground — it reduces overhead while still protecting the data that matters most.

---

## Encrypting Data in Transit

Data traveling across a network must be protected against interception.

### HTTPS

Standard web browsing uses **HTTPS**, which encrypts all traffic between a browser and a web server.

### VPN (Virtual Private Network)

Creates an **encrypted tunnel** between two endpoints, protecting all data passing through it.

| Use Case | Common Protocol |
|----------|----------------|
| Remote access (client-based) | **SSL / TLS** |
| Site-to-site connectivity | **IPsec** |

---

## Encryption Algorithms

An encryption algorithm is the mathematical formula used to both encrypt and decrypt data. Both parties in a communication must use **compatible algorithms**.

### Algorithm Compatibility

- Both sides must agree on the same algorithm before communicating
- Encrypting with one algorithm (e.g., **DES**) and decrypting with another (e.g., **AES**) will fail

### Common Algorithms

| Algorithm | Full Name | Notes |
|-----------|-----------|-------|
| **DES** | Data Encryption Standard | Older; considered weak by modern standards |
| **AES** | Advanced Encryption Standard | Modern standard; multiple key length variants |

### Transparency of Algorithms

Encryption algorithms are typically **public knowledge** — the math and code behind them can be reviewed by anyone. This openness actually increases trust, because the algorithm can be independently verified.

> **The security lies in the key, not the algorithm.** Knowing how an algorithm works does not allow decryption without the key — similar to knowing how a door lock is manufactured but not having the key to open it.

---

## Key Security

### Keeping Keys Private

If an attacker gains access to your private key, they can decrypt all data protected by it. Private keys must be stored securely and never shared unnecessarily.

### Brute Force Attacks

Attackers may attempt to discover a key by trying every possible combination. Longer keys make this exponentially harder.

**Recommended minimum key lengths:**

| Encryption Type | Minimum Key Length |
|----------------|-------------------|
| Symmetric (e.g., AES) | **128 bits** |
| Asymmetric (e.g., RSA) | **3072 bits** |

> As processors become more powerful, minimum recommended key lengths will continue to increase.

### Key Stretching (Key Strengthening)

A technique that applies the encryption or hashing process **multiple times** to the same data, making brute force attacks significantly more time-consuming.

**Example with password hashing:**

```
password → hash → hash(hash) → hash(hash(hash)) → ...
```

Each additional iteration multiplies the time required for a brute force attempt, adding meaningful resistance without changing the underlying algorithm.

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Data at rest** | Use BitLocker, FileVault, or EFS depending on OS and scope |
| **Database encryption** | Choose full or column-level based on performance needs |
| **Data in transit** | Use HTTPS for web traffic; VPNs for site-to-site or remote access |
| **Algorithm compatibility** | Both parties must use the same encryption algorithm |
| **Key length** | Longer keys resist brute force; 128-bit+ for symmetric, 3072-bit+ for asymmetric |
| **Key stretching** | Re-hashing multiple times increases brute force resistance |
