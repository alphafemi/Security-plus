# Digital Certificates & Certificate Authorities

## What Is a Digital Certificate?

A **digital certificate** is a file containing a **public key** and a **digital signature**. Think of it as a digital ID card — but one that also enables encryption, authentication, and trust verification.

Certificates follow the **X.509 standard**, a universal format that allows any browser, operating system, or application to read and validate them consistently.

### What's Inside a Certificate

| Field | Description |
|-------|-------------|
| **Serial number** | Unique identifier for this certificate |
| **Version** | X.509 format version |
| **Signature algorithm** | Algorithm used to create the digital signature |
| **Issuer** | The Certificate Authority that signed it |
| **Subject** | The entity (person, server, org) the cert belongs to |
| **Public key** | The public key associated with the subject |
| **Subject Alternative Name** | Additional domains or hostnames covered by the cert |
| **CRL Distribution Points** | Where to find the list of revoked certificates |
| **Validity period** | The date range during which the cert is valid |

## Trust & Certificate Authorities

### The Problem of Trust

When a browser connects to a website for the first time, how does it know the site is legitimate? It has no prior relationship with that server. The answer lies in a system of **trusted third parties**.

### Certificate Authorities (CAs)

A **Certificate Authority (CA)** is a trusted organization that:

1. **Validates** that an applicant truly owns or controls the resource they're requesting a certificate for
2. **Digitally signs** the certificate using the CA's private key
3. **Vouches** for the certificate holder — if your browser trusts the CA, it will trust any certificate the CA has signed

Browsers ship with a built-in list of **hundreds of trusted CAs**. Any website that holds a certificate signed by one of these CAs will be automatically trusted.

> The value of a CA is not just the digital signature — it's the **validation process** behind it. Without that verification, the signature means nothing.

### Root of Trust

The **root of trust** is the foundational component from which all other trust is derived. In browsers, this is the pre-installed list of trusted CAs. On devices, it may be a Hardware Security Module, Secure Enclave, or a TPM — whatever anchors the chain of trust for that system.

### Web of Trust

An alternative to centralized CAs is the **web of trust** model, where individuals sign each other's certificates directly. If you trust Alice, and Alice has signed Bob's certificate, you can extend that trust to Bob. This decentralized model is used in systems like PGP/GPG.

## Obtaining a Certificate: The CSR Process

To get a publicly trusted certificate for a web server, the process works as follows:

1. **Generate a key pair** — create a public/private key pair for your server
2. **Create a Certificate Signing Request (CSR)** — combine your public key with identifying information (domain name, organization details)
3. **Submit the CSR to a CA** — send the request to a certificate authority for validation
4. **CA validates ownership** — the CA confirms you own and control the domain or resource
5. **CA signs and returns the certificate** — the CA uses its private key to digitally sign your certificate and returns it
6. **Install the certificate** — deploy the signed certificate on your web server

```
[Your Server]                          [Certificate Authority]
     |                                         |
     |── CSR (public key + identity) ────────► |
     |                                         | (validates ownership)
     |◄── Signed Certificate ─────────────────-|
     |                                         |
     | (install on server)                     |
```

## Internal / Private Certificate Authorities

If your organization only needs certificates for internal services, you can **run your own CA** rather than purchasing certificates from a public one.

### How It Works

1. Install CA software internally (e.g., **Microsoft Windows Certificate Services**, **OpenCA**, or other third-party tools)
2. Distribute your internal CA's public certificate to all devices in the organization
3. All devices will now trust certificates signed by your internal CA, just as they would a public CA

### When to Use an Internal CA

| Scenario | Recommended CA |
|----------|---------------|
| Public-facing website | Public CA (browser-trusted) |
| Internal applications and services | Internal CA |
| Development and testing | Internal CA or self-signed |

## Wildcard Certificates & Subject Alternative Names

A **Subject Alternative Name (SAN)** allows a single certificate to cover multiple hostnames or domains. When a wildcard (`*`) is used, the certificate covers all subdomains matching that pattern.

**Example:**

```
*.birdfeeder.live
```

This single certificate is valid for:
- `www.birdfeeder.live`
- `mail.birdfeeder.live`
- `ftp.birdfeeder.live`
- Any other subdomain under `birdfeeder.live`

This makes certificate management significantly easier for large organizations — one certificate can cover an entire fleet of servers sharing a domain.

## Certificate Revocation

Certificates sometimes need to be invalidated before they expire — for example, when a server is decommissioned, or when a private key is compromised.

### Real-World Example: Heartbleed (2014)

The **Heartbleed** vulnerability in the OpenSSL library allowed attackers to extract a web server's private key. Once discovered, organizations had to revoke all affected certificates and issue new ones. This highlighted the critical importance of having a reliable revocation mechanism.

---

### Method 1: Certificate Revocation List (CRL)

A **CRL** is a file published by a CA that lists all certificates it has revoked.

**Browser process:**
1. Connect to a website and receive its certificate
2. Find the **CRL Distribution Points** field in the certificate
3. Download the CRL from the listed URI
4. Check whether the certificate appears in the list
5. If listed → reject the connection. If not listed → proceed.

**Limitation:** Downloading a full CRL for every connection is inefficient, especially as the list grows.

---

### Method 2: OCSP — Online Certificate Status Protocol

**OCSP** provides a more efficient alternative to CRL downloads. Instead of downloading a full list, a browser queries a server for the status of a single specific certificate.

#### OCSP Stapling

To avoid relying on a separate third-party OCSP server, the web server itself can include a **pre-fetched, CA-signed OCSP status response** directly in the SSL/TLS handshake. This is called **OCSP stapling**.

```
Standard OCSP:
Browser ──► Web Server (get cert) ──► CA's OCSP Server (check status) ──► Browser

OCSP Stapling:
Browser ──► Web Server (get cert + CA-signed status in handshake) ──► Browser
```

**Benefits of OCSP Stapling:**
- Faster — no additional round trip to a third-party server
- More private — the CA's OCSP server doesn't see every browser request
- The CA's digital signature on the status response ensures it cannot be forged

---

## Revocation Method Comparison

| Method | How It Works | Efficiency | Notes |
|--------|-------------|------------|-------|
| **CRL** | Browser downloads a full list of revoked certs | Low — full file download per check | Becomes unwieldy as list grows |
| **OCSP** | Browser queries status of a single cert | Medium — one request per cert | Requires third-party OCSP server |
| **OCSP Stapling** | Server embeds CA-signed status in handshake | High — no extra round trip | Preferred modern approach |

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Digital certificate** | Contains a public key and digital signature; follows X.509 format |
| **Certificate Authority (CA)** | Validates identity and digitally signs certificates to establish trust |
| **Root of trust** | The foundational trusted component from which all other trust is derived |
| **Web of trust** | Decentralized trust model where individuals sign each other's certificates |
| **CSR** | A certificate signing request sent to a CA to obtain a signed certificate |
| **Internal CA** | An organization-run CA for issuing certificates to internal services |
| **Wildcard / SAN certificate** | A single certificate valid for multiple subdomains |
| **CRL** | A CA-published list of revoked certificates |
| **OCSP** | Protocol for checking the revocation status of a single certificate in real time |
| **OCSP Stapling** | CA-signed status embedded in the TLS handshake for efficient revocation checking |
