# Key Exchange

## The Challenge

For encryption to work, both parties must share the same key — but transmitting that key over an insecure medium like the internet creates a fundamental risk: if an attacker intercepts the key, they can decrypt everything protected by it.

This page covers the main approaches to solving this problem.

---

## Out-of-Band Key Exchange

**Out-of-band** key exchange means transferring the key through a channel that is completely separate from the network being secured.

Common methods include:

- Physical courier (e.g., a locked briefcase hand-delivered between parties)
- In-person exchange
- Phone call

**Limitation:** Out-of-band exchange is impractical for real-time internet communication. There is no time to physically transfer a key before a browser session, for example. It works only when both parties can plan ahead and are not time-constrained.

---

## In-Band Key Exchange

**In-band** key exchange sends key-related information across the same network being used for communication. Since sending a raw key over an insecure network is dangerous, additional techniques are required to protect it.

### Asymmetric Encryption of a Symmetric Key

One common approach is to wrap a symmetric key inside asymmetric encryption before transmitting it:

1. The client generates a symmetric key (e.g., a session key).
2. The client encrypts that symmetric key using the **server's public key**.
3. The encrypted key is sent across the network.
4. The server decrypts it using its **private key**, recovering the symmetric key.
5. Both sides now share the same symmetric key and can communicate securely.

Because the symmetric key is encrypted with asymmetric encryption during transfer, intercepting the transmission reveals nothing useful without the server's private key.

### Session Keys

Session keys are a common application of this pattern. They are:

- **Ephemeral (temporary)** — used only for the duration of a single session
- **Discarded** after the session ends
- **Regenerated** for each new session

Using short-lived session keys limits the damage if a key is ever compromised — past sessions remain protected even if a future key is exposed.

---

## Key Exchange Algorithms

A more elegant solution is to have both parties **independently derive the same symmetric key** without ever transmitting it across the network.

### How It Works

This approach relies on the mathematical relationship between public and private key pairs. Using a key exchange algorithm:

- **Bob** combines his **private key** with **Alice's public key**
- **Alice** combines her **private key** with **Bob's public key**

Because both key pairs are mathematically related, both operations produce **the same symmetric key** — even though neither party sent that key to the other.

```
Bob:   Bob's Private Key  +  Alice's Public Key  →  Shared Symmetric Key
Alice: Alice's Private Key +  Bob's Public Key   →  Shared Symmetric Key
```

> **Important:** This process is not encryption or hashing. It is purely a method for constructing a shared key on both sides of a conversation without transmitting it.

### Why This Is Secure

- Alice's public key is freely available — no secret information is transmitted
- Bob's public key is freely available — same applies
- The private keys **never leave their respective owners**
- An attacker who intercepts the public keys cannot reconstruct the shared symmetric key without a private key

---

## Comparison

| Method | Key Sent Over Network? | Practical for Real-Time Use? | Notes |
|--------|----------------------|------------------------------|-------|
| **Out-of-band** | No | No | Requires physical coordination |
| **Asymmetric-wrapped symmetric** | Yes (encrypted) | Yes | Relies on server's public/private key pair |
| **Key exchange algorithm** | No | Yes | Derives shared key independently on each side |

---

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Out-of-band exchange** | Secure but impractical for real-time internet use |
| **In-band exchange** | Key material is sent over the network, protected by asymmetric encryption |
| **Session keys** | Temporary, ephemeral keys that limit exposure if compromised |
| **Key exchange algorithms** | Both parties derive the same key independently — nothing sensitive is transmitted |
