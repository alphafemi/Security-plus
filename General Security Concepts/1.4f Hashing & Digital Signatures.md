# Hashing & Digital Signatures

## What Is a Cryptographic Hash?

A **cryptographic hash** converts an input of any size into a fixed-length string of characters, sometimes called a **message digest** or **fingerprint**. Like a fingerprint, a hash represents the original data without containing it — you cannot reconstruct the original data from the hash alone.

> **Hashing is not encryption.** There is no key, and the process is one-way. Given only a hash, the original input cannot be recovered.

### Properties of a Good Hash

- **Deterministic** — the same input always produces the same output
- **Unique** — different inputs should always produce different outputs
- **Sensitive** — even a single character change produces a completely different hash
- **Non-reversible** — the original input cannot be derived from the hash

### Example: SHA-256

SHA-256 produces a 256-bit output represented as 64 hexadecimal characters. A one-character difference in input produces a completely different hash:

| Input | SHA-256 Hash |
|-------|-------------|
| `My name is Professor Messer.` | `e3b0c4...` |
| `My name is Professor Messer!` | `7f83b1...` |

## Hash Collisions

A **collision** occurs when two different inputs produce the same hash output. This should be an extremely rare or impossible occurrence in a well-designed algorithm.

### MD5 and the Collision Problem

The **MD5** hashing algorithm was found to have a collision vulnerability in **1996**. It is possible to craft two different inputs that produce identical MD5 hashes, completely undermining the integrity guarantees of the algorithm.

> ⚠️ **MD5 should not be used** for any security-sensitive purpose. Use SHA-256 or a more modern algorithm instead.

## Common Uses of Hashing

### 1. File Integrity Verification

When downloading files (e.g., a Linux ISO), websites often publish the expected hash alongside the download. After downloading, you run the same hashing algorithm on your local file and compare the result to the published hash.

- **Hashes match** → the file arrived intact and unmodified
- **Hashes differ** → the file was corrupted or tampered with during download

### 2. Password Storage

Passwords should never be stored as plain text, and they should not be encrypted (which could be reversed). Instead, passwords are stored as hashes.

**Login process:**
1. User enters password
2. System hashes the input
3. The resulting hash is compared to the stored hash
4. If they match, access is granted — the system never needs to know the actual password

#### Salted Hashes

A **salt** is a random value added to a password before hashing. Each user receives a unique salt, which means:

- Two users with the same password will have different stored hashes
- Pre-computed attack tables become ineffective

**Example — same password, different salts:**

| Password | Salt | Resulting Hash |
|----------|------|---------------|
| `dragon` | *(none)* | `a1b2c3...` |
| `dragon` | `x9f2` | `7e4d91...` |
| `dragon` | `m3k7` | `c820fa...` |
| `dragon` | `q1r5` | `3bf741...` |

Even though all four rows use the same password, an attacker viewing the hash database would assume four different passwords were in use.

#### Rainbow Tables

A **rainbow table** is a pre-computed lookup table mapping common inputs to their hash values, allowing near-instant reversal of unsalted hashes. Salting defeats rainbow tables entirely — the attacker must resort to brute force, which can take days, weeks, or longer.

## Digital Signatures

A **digital signature** is a cryptographic mechanism that provides three guarantees:

| Property | What It Proves |
|----------|---------------|
| **Integrity** | The message was not altered after signing |
| **Authentication** | The message came from the stated sender |
| **Non-repudiation** | The sender cannot deny having sent the message |

> Digital signatures use the **opposite** key from encryption. A message is signed with the sender's **private key** and verified with their **public key**.

---

### How Digital Signatures Work

#### Signing (Sender — Alice)

1. Alice writes her plaintext message: `"You're hired, Bob."`
2. Her email client runs the plaintext through a **hashing algorithm**, producing a hash
3. That hash is **encrypted with Alice's private key** — this is the digital signature
4. The plaintext and the digital signature are sent together to Bob

```
Plaintext ──► [Hash Algorithm] ──► Hash ──► [Encrypt with Alice's Private Key] ──► Digital Signature
```

#### Verifying (Recipient — Bob)

1. Bob receives the plaintext and the digital signature
2. His email client **decrypts the digital signature using Alice's public key**, recovering the original hash
3. His client independently **hashes the received plaintext**
4. The two hashes are compared

```
Digital Signature ──► [Decrypt with Alice's Public Key] ──► Hash A
Received Plaintext ──► [Hash Algorithm] ──────────────────► Hash B

Hash A == Hash B  →  ✅ Signature verified: message is authentic and unmodified
Hash A != Hash B  →  ❌ Verification failed: message was altered or signature is invalid
```

> **Note:** In this process, the message itself is sent as plaintext — digital signatures provide integrity and authentication, not confidentiality. To also protect the content, the message would need to be separately encrypted.

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Cryptographic hash** | One-way fingerprint of data; cannot be reversed to recover the original |
| **SHA-256** | Modern, widely used hashing algorithm producing a 64-character hex output |
| **Collision** | When two different inputs produce the same hash; indicates a weak algorithm |
| **MD5** | Deprecated due to known collision vulnerabilities |
| **Password hashing** | Passwords are stored as hashes, never plain text or encrypted |
| **Salt** | Random value added before hashing to prevent rainbow table attacks |
| **Rainbow table** | Pre-computed hash lookup — defeated by salting |
| **Digital signature** | Signed with private key, verified with public key; provides integrity, authentication, and non-repudiation |
