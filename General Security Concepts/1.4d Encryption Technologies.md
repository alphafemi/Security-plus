# Cryptographic Hardware & Key Management

## Overview

As encryption is applied across more devices, users, and services, managing cryptographic functions and keys at scale requires dedicated hardware and centralized software. This page covers the main technologies used to meet that need.

---

## Trusted Platform Module (TPM)

A **Trusted Platform Module** is a standardized chip or subsystem found on modern motherboards, designed to provide cryptographic functions for a single device.

### Capabilities

- **Random number and key generation** — produces cryptographic material in hardware
- **Persistent memory** — keys can be burned into the TPM and remain unique to that machine
- **Secure key storage** — protects keys locally without exposing them to the OS
- **Brute force resistance** — TPM-stored data cannot be attacked via brute force or dictionary attacks

### Common Use Case

TPMs are commonly used with full-disk encryption tools like **BitLocker**, where the TPM generates and stores the encryption keys tied to a specific machine. This ensures the drive cannot be decrypted if removed and placed in a different system.

> **Scope:** One TPM serves one device.

---

## Hardware Security Module (HSM)

A **Hardware Security Module** provides the same cryptographic functions as a TPM but at data center scale — serving hundreds or thousands of devices from a centralized unit.

### Capabilities

- **Centralized key storage** — stores encryption keys for large numbers of servers and services
- **High availability** — HSMs in enterprise environments are clustered with redundant power supplies and network connectivity
- **Hardware-accelerated cryptography** — dedicated plug-in cards or co-processors perform encryption and decryption in real time
- **Unauthorized access prevention** — keys are stored securely and cannot be extracted without authorization

### Common Use Case

A data center running a thousand web servers can store all TLS/SSL keys on a centralized HSM rather than distributing them across individual machines. Cryptographic accelerators on the HSM handle encryption and decryption at the speed required for large-scale, real-time traffic.

> **Scope:** One HSM cluster serves an entire data center or organization.

---

## TPM vs. HSM

| Feature | TPM | HSM |
|---------|-----|-----|
| **Scope** | Single device | Enterprise / data center |
| **Location** | Onboard motherboard chip | Dedicated appliance or cluster |
| **Redundancy** | None | Clustered with failover |
| **Use case** | Full-disk encryption, device-level keys | Web servers, PKI, large-scale key storage |
| **Cryptographic acceleration** | Limited | Yes, via dedicated hardware |

## Key Management Systems

With encryption keys spread across web servers, individual devices, user certificates, and more, a **centralized key management system** provides a single console to manage all of them.

### Deployment Options

- **On-premises** — runs on hardware within your own infrastructure
- **Cloud-based** — accessible from anywhere; managed externally

### Features

- **Unified management** — handles SSL/TLS, SSH, BitLocker, Active Directory, and other key types from one interface
- **Separation of keys and data** — keys are stored independently from the data they protect
- **User association** — keys are linked to specific users or services within the system
- **Automatic key rotation** — keys are cycled on a schedule to reduce long-term exposure
- **Logging and reporting** — tracks key usage, active vs. inactive keys, expiration dates, and certificate authority details

### Dashboard View

A typical key management dashboard provides:

- Summary of key types in use
- Certificate authority information and certificate expiration dates
- SSL/TLS key inventory with associated servers
- SSH key fingerprints and usage locations
- License details and compliance reporting

## Secure Enclaves

A **secure enclave** is a dedicated security processor built into a device — separate from the main CPU — whose sole purpose is protecting data privacy.

Secure enclaves are found in mobile phones, laptops, and desktop systems. Different manufacturers use different names for this component, but the function is the same.

### Capabilities

| Feature | Description |
|---------|-------------|
| **Separate boot ROM** | Boots independently of the main OS |
| **Process monitoring** | Monitors system processes, especially during boot |
| **True random number generator** | Produces cryptographically secure randomness |
| **Real-time memory encryption** | Encrypts data as it moves in and out of memory |
| **Immutable cryptographic keys** | Built-in root keys that cannot be changed |
| **Hardware AES encryption** | Performs AES in hardware for speed and isolation |

### Why It Matters

Even if a device falls into the wrong hands, the secure enclave ensures that data remains private. Because it operates independently of the main processor and OS, compromising the OS does not compromise the enclave.

> The secure enclave acts as a **root of trust** — the foundation from which all other cryptographic operations on the device are derived.

## Summary

| Technology | Primary Purpose | Typical Scale |
|-----------|----------------|---------------|
| **TPM** | Device-level cryptographic functions and key storage | Single machine |
| **HSM** | Enterprise-scale key storage and cryptographic acceleration | Data center |
| **Key Management System** | Centralized administration of all key types | Organization-wide |
| **Secure Enclave** | Hardware-isolated data privacy and root of trust | Single device |
