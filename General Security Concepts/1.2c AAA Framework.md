## Overview

The **AAA framework** is a security model that defines how to:

1. **Verify identity** - Know who is accessing resources
2. **Control access** - Allow only appropriate access
3. **Record activity** - Maintain audit trails for compliance and security

### Why AAA Matters

- **Security** - Prevents unauthorized access to systems and data
- **Compliance** - Meets regulatory requirements (HIPAA, GDPR, SOX, etc.)
- **Auditability** - Tracks who did what and when
- **Accountability** - Creates responsibility for actions
- **Scalability** - Manages access for large numbers of users and resources

### AAA in Daily Life

Every time you:
- Log into your computer with username and password
- Access files you're permitted to view
- Have your login recorded in the system log

You're using the AAA framework.

## The Three Pillars of AAA

### Authentication (A)

**Definition:** The process of verifying that someone is who they claim to be.

**The Question:** "Are you really who you say you are?"

**How it works:**
- User claims identity (identification)
- System verifies claim using credentials
- Credentials prove the user's identity

**Authentication Factors:**

| Factor | What You | Example |
|--------|----------|---------|
| **Something you know** | Knowledge | Password, PIN, security question |
| **Something you have** | Possession | Hardware token, phone, security key |
| **Something you are** | Inherence | Fingerprint, facial recognition, iris scan |
| **Somewhere you are** | Location | GPS location, network location |

**Multi-Factor Authentication (MFA):**
- Requires two or more factors
- Much more secure than single factor
- Even if password is compromised, account remains protected
- Example: Password + SMS code to phone

**Examples of Authentication:**
- Username and password login
- Biometric fingerprint scan
- Hardware security key
- One-time password (OTP)
- Digital certificate validation
- Challenge-response questions

**Key Point:** Authentication proves identity; it does NOT grant access.

---

### Authorization (A)

**Definition:** The process of determining what an authenticated user is allowed to do and access.

**The Question:** "Now that I know who you are, what are you allowed to access?"

**How it works:**
- User is authenticated (proven to be who they claim)
- System determines what resources user can access
- Access is granted or denied based on permissions
- User can only perform authorized actions

**Authorization Decisions:**

| Decision | Meaning | Example |
|----------|---------|---------|
| **Allow** | User can access resource | Jane from Finance can view financial reports |
| **Deny** | User cannot access resource | Jane from Finance cannot access employee personal records |
| **Conditional** | Access allowed with conditions | Alice can create reports only during business hours |

**What Authorization Controls:**
- Which files a user can read, write, delete
- Which applications a user can run
- Which databases a user can query
- Which functions a user can perform
- Which data a user can see

**Key Principle:** Authorization ensures the principle of least privilege—users get minimum access needed to do their job.

**Key Point:** Authorization defines what authenticated users can do.

---

### Accounting (A)

**Definition:** The process of recording and tracking user activity and resource usage.

**The Question:** "What did this user do, when did they do it, and how much of our resources did they use?"

**How it works:**
- Activities are logged when they occur
- Logs capture who, what, when, where
- Logs are stored for later review and analysis
- Logs support compliance and forensics

**What Accounting Records:**

| Information | Purpose | Example |
|-------------|---------|---------|
| **Who** | Identity of user | username: jsmith |
| **What** | Action performed | action: logged in, viewed file, deleted email |
| **When** | Timestamp | 2024-04-30 14:35:22 UTC |
| **Where** | Source location | source_ip: 192.168.1.100 |
| **Duration** | Time connected | session_duration: 2 hours 15 minutes |
| **Data volume** | Resources used | bytes_transferred: 1.2 GB, queries: 543 |
| **Result** | Success/failure | status: success, failure_reason: access_denied |

**Accounting Use Cases:**

1. **Security Monitoring** - Detect unusual activity patterns
2. **Forensics** - Investigate security incidents
3. **Compliance** - Prove regulatory compliance
4. **Performance** - Analyze resource usage
5. **Billing** - Charge users for resource consumption
6. **Accountability** - Hold users responsible for actions

**Log Analysis Examples:**
```
Failed login attempts:
User: jsmith | 3 attempts in 5 minutes | IP: 203.0.113.45
→ Possible brute force attack

Unusual file access:
User: mtaylor | 500 GB downloaded at 2:00 AM | IP: 10.0.0.50
→ Possible data exfiltration

Resource usage spike:
User: apatterson | 1000 database queries in 1 hour | Time: 15:30-16:30
→ Possible malicious query or legitimate data pull
```

**Key Point:** Accounting creates the audit trail for compliance, forensics, and security monitoring.

## Identification vs. Authentication

### Important Distinction

| Aspect | Identification | Authentication |
|--------|---|---|
| **Definition** | Claiming to be a user | Proving you are that user |
| **Question** | "Who do you claim to be?" | "Prove you are who you claim" |
| **Input** | Username, email, ID | Password, biometric, certificate |
| **Verification** | None (just a claim) | Cryptographic or biological verification |
| **Security** | Alone: no security | Provides actual security |
| **Example** | "I am jsmith" | "I know the password" or "my fingerprint matches" |

### The Process

```
User enters username: "jsmith"
        ↓
IDENTIFICATION CLAIMED (no verification yet)
        ↓
System prompts: "Enter password"
        ↓
User enters password: "correct_password"
        ↓
AUTHENTICATION (system verifies the claim)
        ↓
Password check: ✓ MATCH
        ↓
Result: Identity confirmed - User IS jsmith
```

**Key Point:** Identification is claiming an identity; authentication is proving it.

## VPN Authentication Example

### Real-World Scenario: Remote Employee Accessing Company Network

**Situation:** A remote employee needs to access internal company systems from home.

### Step 1: Client Initiates Connection

```
Remote Employee (Home)          VPN Concentrator (Company Network)
        ↓                                    ↓
    Opens VPN client              Waits for connection request
    Enters username: jsmith       ←──────────┤
    Enters password               │ Request received
    Clicks "Connect"              │
        ↓                         ↓
    Credentials sent to concentrator
        │─────────────────────────→
                                  Receives credentials
                                  (doesn't validate locally)
```

### Step 2: VPN Concentrator Queries AAA Server

```
VPN Concentrator                    AAA Server
        │                                ↓
        │ "Does user 'jsmith' exist?"   │
        │ "Is their password correct?"  │
        │─────────────────────────────→ Checks database
        │                               Looks up jsmith
        │                               Verifies password
        │                               Generates response
        │
    Response received: "APPROVED"
        │←────────────────────────────
        ↓
    Status: User authenticated
```

**Why Use AAA Server?**
- **Centralized management** - One place to manage all users
- **Scalability** - Thousands of VPN concentrators can query same server
- **Security** - VPN concentrator doesn't store passwords
- **Consistency** - All systems use same credentials
- **Auditability** - AAA server logs all authentication attempts

### Step 3: Authentication Approved

```
AAA Server confirms: Username and password match ✓
        ↓
VPN Concentrator: "User authenticated successfully"
        ↓
Determine authorization: User is member of "Remote Workers" group
        ↓
"Remote Workers" group permissions:
  ✓ Can access internal file server
  ✓ Can access email system
  ✗ Cannot access HR database
  ✗ Cannot access financial systems
        ↓
VPN connection established
        ↓
Log entry created:
  User: jsmith | Action: VPN login | Time: 14:35:22 | IP: 203.0.113.100 | Status: success
```

### Step 4: User Accesses Resources

```
User connects to file server
        ↓
File server checks:
  Authorization: jsmith is in "Remote Workers" group ✓
  File: Can Remote Workers access this file? YES ✓
        ↓
File access granted
        ↓
Log entry created:
  User: jsmith | Action: opened file payroll.xlsx | Time: 14:38:10 | Size: 2.1 MB
```

### Step 5: User Logs Off

```
User clicks "Disconnect" on VPN client
        ↓
VPN Concentrator terminates connection
        ↓
Session ends: 14:35:22 - 15:20:45 (45 minutes, 125 MB transferred)
        ↓
Log entry created:
  User: jsmith | Action: VPN logout | Time: 15:20:45 | Duration: 45 min | Bytes: 125 MB
```

## Device Authentication

### Challenge: Authenticating Computers

**Problem:** How do you authenticate a computer? A laptop can't type a password.

**Traditional Approach:**
- Store password on laptop
- Laptop logs in automatically
- **Problem:** If laptop is stolen, attacker has credentials

**Solution:** Use Digital Certificates

### How Certificate-Based Device Authentication Works

### Step 1: Create Certificate Authority (CA)

```
Organization Sets Up Security Infrastructure
        ↓
Create Certificate Authority (CA)
  - Device that manages certificates
  - Has its own certificate (self-signed)
  - Digitally signs all device certificates
  - Trusted root in security infrastructure
```

### Step 2: Create Device Certificate

```
Administrator needs to authenticate a company laptop
        ↓
Contact Certificate Authority
        ↓
CA generates device certificate containing:
  - Device identity (Asset tag: LT-001234)
  - Public key for this device
  - Certificate validity period
  - Device serial number
  - Organizational unit
        ↓
CA digitally signs this certificate with CA's private key
        ↓
Signed device certificate is installed on laptop
```

### Step 3: Present Certificate for Authentication

```
Company Laptop              VPN Concentrator
        │                           ↓
        │ "I am device LT-001234"  │
        │ (presents certificate)    │
        │───────────────────────→
                              Receives certificate
                              Checks certificate signature
                              Verifies CA signature
                              Checks certificate expiration
                              Matches certificate against
                              trusted CA certificates
        │
        │←────────────────────────
        │  "Certificate verified"
        │  "You are authenticated"
        ↓
VPN access granted
```

### Step 4: Verify Certificate Chain

```
Device Certificate
        ↓
Signed by: CA Certificate
        ↓
CA Certificate is signed by: Root CA Certificate
        ↓
Root CA Certificate exists in our trusted store
        ↓
Result: Chain of trust verified ✓
        ↓
Device is company-owned and authenticated
```

**Benefits of Certificate-Based Device Authentication:**

✅ **No passwords to steal** - Certificate-based, not password-based
✅ **Company-owned verification** - Proves device is company equipment
✅ **Scalable** - Works for thousands of devices
✅ **Automatic** - User doesn't enter credentials
✅ **Revocable** - Can disable compromised certificates
✅ **Audit trail** - Logs all authentication attempts

## Authorization Models

### Challenge: Scaling Authorization

**Simple Approach: Direct Permissions**

```
User: jsmith
  ✓ Can create shipping labels
  ✓ Can track shipments
  ✓ Can view monthly reports
  ✓ Can access customer data

User: apatterson
  ✓ Can create shipping labels
  ✓ Can track shipments
  ✓ Can view monthly reports
  ✓ Can access customer data

User: mchen
  ✓ Can create shipping labels
  ✓ Can track shipments
  ✓ Can view monthly reports
  ✓ Can access customer data

[Repeat for 100+ users]
```

**Problems:**
- ❌ Manual assignment to each permission for each user
- ❌ Time-consuming to set up
- ❌ Easy to make mistakes (missing permissions)
- ❌ Hard to change (must update every user if rules change)
- ❌ Doesn't scale (imagine 1000 users and 100 resources)

### Solution: Authorization Models with Abstraction

Instead of assigning permissions directly to users, use an abstraction layer:

```
Users → Groups → Permissions → Resources
```

---

### Discretionary Access Control (DAC)

**Definition:** Resource owner decides who gets access and what they can do.

**How it works:**
- Resource owner (like a manager) sets permissions
- Owner determines who can read, write, delete
- Permissions are discretionary (owner's choice)

**Example:**
```
File: Q4_Budget.xlsx
Owner: Finance Director
Permissions set by owner:
  ✓ Read: Finance team members
  ✓ Write: Finance director, budget analyst
  ✗ Delete: No one (document retention policy)
```

**Advantages:**
- Flexible - Owner controls who gets access
- Simple - Easy to understand
- Responsive - Can change quickly

**Disadvantages:**
- Less secure - Users may not manage permissions properly
- Inconsistent - Different owners might have different policies
- Not scalable - Manual permission assignment

---

### Role-Based Access Control (RBAC)

**Definition:** Permissions are assigned to roles, and users are assigned to roles.

**How it works:**
- Identify job roles in organization
- Define what each role needs to access
- Assign users to roles
- Users inherit all permissions for their role

**Example: Shipping & Receiving Department**

```
Define Role: Shipping & Receiving Employee
  Permissions for this role:
    ✓ Create shipping labels
    ✓ Track shipments
    ✓ View monthly shipment reports
    ✓ Access customer contact information
    
Now add users to this role:
  jsmith → Add to "Shipping & Receiving Employee" role
  apatterson → Add to "Shipping & Receiving Employee" role
  mchen → Add to "Shipping & Receiving Employee" role
  [100+ more users]

Each user automatically gets ALL permissions for the role
```

**Matrix View:**

```
                        Create Labels  Track Shipments  View Reports  Customer Data
jsmith                       ✓              ✓              ✓             ✓
apatterson                   ✓              ✓              ✓             ✓
mchen                        ✓              ✓              ✓             ✓
[100+ Shipping users]        ✓              ✓              ✓             ✓

Shipping Supervisor
  Role: Supervisor
  Permissions:         ✓              ✓              ✓             ✓
  Additional:                                         ✓ Generate reports
                                                      ✓ Manage team
```

**Advantages:**
- **Scalable** - One role definition covers many users
- **Maintainable** - Change role once affects all users with that role
- **Consistent** - All shipping employees have same permissions
- **Simple** - Easy to understand and manage

**Disadvantages:**
- **Coarse-grained** - Difficult to grant different permissions to users in same role
- **Role explosion** - Need many roles to cover all scenarios

---

### Attribute-Based Access Control (ABAC)

**Definition:** Access decisions based on attributes of user, resource, environment, and action.

**How it works:**
- Evaluate multiple attributes
- Make access decision based on combination
- Very flexible and fine-grained

**Attributes:**

| Attribute Type | Examples |
|---|---|
| **User attributes** | Department, role, title, location, security clearance |
| **Resource attributes** | Classification level, owner, type, age, sensitivity |
| **Action attributes** | Read, write, delete, create, export |
| **Environment attributes** | Time of day, network location, device type, authentication strength |

**Policy Example:**

```
Allow access if:
  User.Department = "Finance" AND
  Resource.Classification = "Financial" AND
  User.AuthenticationStrength >= "MFA" AND
  Environment.TimeOfDay = "Business Hours" AND
  Environment.DeviceType = "Company-owned"
```

**Scenarios:**

```
Scenario 1: John from Finance, MFA, company laptop, Tuesday 10 AM
  ✓ Access to Financial report GRANTED

Scenario 2: John from Finance, password only, personal laptop, Tuesday 10 AM
  ✗ Access to Financial report DENIED (not MFA, personal device)

Scenario 3: John from Finance, MFA, company laptop, Sunday 2 AM
  ✗ Access to Financial report DENIED (outside business hours)

Scenario 4: Sarah from Marketing, MFA, company laptop, Tuesday 10 AM
  ✗ Access to Financial report DENIED (not Finance department)
```

**Advantages:**
- **Fine-grained** - Very specific access control
- **Flexible** - Handles complex scenarios
- **Dynamic** - Can include environmental factors
- **Powerful** - Supports advanced security policies

**Disadvantages:**
- **Complex** - Difficult to design and manage
- **Computational cost** - More evaluation overhead
- **Testing challenge** - Hard to test all combinations

---

### Organization-Based Access Control

**Definition:** Access based on organizational hierarchy and structure.

**How it works:**
- Organize users by department, team, location
- Define access by organizational unit
- Users inherit permissions from organization level

**Example:**

```
Organization Structure:
  Company
    ├─ Finance Department
    │   ├─ Finance Team (Group)
    │   └─ Accounting Team (Group)
    ├─ Shipping & Receiving
    │   ├─ Warehouse Team (Group)
    │   └─ Logistics Team (Group)
    └─ IT Department
        ├─ Systems Team (Group)
        └─ Security Team (Group)

Permissions assigned:
  Finance Department
    ✓ Can access financial systems
    ✓ Can view budgets
    ✗ Cannot access HR records
    
  Finance Team (within Finance Department)
    ✓ Inherits Finance Department permissions
    ✓ Can create invoices
    ✗ Cannot approve budgets > $100K
    
  Finance Manager (individual override)
    ✓ Inherits all Finance Team permissions
    ✓ Can approve budgets up to $500K
```

**Advantages:**
- **Intuitive** - Follows organization structure
- **Hierarchical** - Permissions inherit from parent org units
- **Maintainable** - Change one org unit's permissions affects all members

## Certificate-Based Authentication

### Why Certificates?

**Scenario:** Managing thousands of systems worldwide

**Challenge:**
- Cannot store passwords on remote systems (security risk)
- Cannot have users manually authenticate each system
- Need automated, scalable authentication

**Solution:** Digital Certificates

### Certificate Components

```
Digital Certificate
├─ Subject (Certificate Holder)
│   ├─ Common Name: LT-001234 (Device identifier)
│   ├─ Organization: ACME Corp
│   └─ Organizational Unit: IT
├─ Public Key
│   └─ Used for encryption and verification
├─ Issuer (Certificate Authority)
│   ├─ Name: ACME Corp CA
│   └─ Signature: CA's digital signature
├─ Validity Period
│   ├─ Valid From: 2024-01-01
│   └─ Valid Until: 2026-01-01
├─ Serial Number: 0x1A2B3C4D
├─ Extensions
│   ├─ Key Usage: Digital Signature
│   ├─ Extended Key Usage: TLS Client Authentication
│   └─ Subject Alternative Name: LT-001234.corp.com
└─ Signature
    └─ Digital signature of CA (proves authenticity)
```

### Certificate Authentication Flow

```
System with Certificate         Authentication Server
        ↓                                 ↓
  Presents certificate           Receives certificate
  (LT-001234)                    ↓
        │                        Verify signature
        │                        (Check CA signed it)
        │                        ↓
        │                        Check expiration
        │←─────────────────────  (Is it still valid?)
        │                        ↓
        │   Certificate Valid    Check revocation list
        │   and Verified         (Has it been revoked?)
        │                        ↓
        │                        Trust the device
        │←─────────────────────  Verify against known good
        │   Authentication       device certificates
        │   Granted              ↓
        ↓                        Check all attributes match
    Access Granted              
```

### Key Advantages

✅ **No passwords needed** - Certificate-based authentication
✅ **Automated** - No user interaction required
✅ **Scalable** - Works for thousands of devices
✅ **Auditable** - All authentication attempts logged
✅ **Revocable** - Can disable compromised certificates
✅ **Secure** - Cryptographically secure verification

## Scaling Authorization

### Problem: Direct User-Permission Model

```
10 Shipping Employees
3 Permissions each
Total: 10 × 3 = 30 mappings

50 Shipping Employees
50 × 3 = 150 mappings

100 Shipping Employees
100 × 3 = 300 mappings

1000 Employees across all departments
1000 × 10 = 10,000 mappings!
```

**Issues:**
- ❌ Error-prone
- ❌ Time-consuming to maintain
- ❌ Hard to audit
- ❌ Difficult to change policies

### Solution: Group-Based Model

```
Create one "Shipping & Receiving Group"
Define permissions once
  ✓ Create shipping labels
  ✓ Track shipments
  ✓ View reports
  ✓ Access customer data

Add users to group (instead of adding permissions):
  jsmith → Shipping & Receiving Group ✓
  apatterson → Shipping & Receiving Group ✓
  mchen → Shipping & Receiving Group ✓
  [1000 more shipping employees] → Group ✓

Result:
  10 users = 10 additions to group
  100 users = 100 additions to group
  1000 users = 1000 additions to group
  (Not 10×3, 100×3, 1000×3 direct permissions)
```

### How Group-Based Authorization Works

```
User authenticates: jsmith logs in
        ↓
System queries: What groups does jsmith belong to?
        ↓
Groups returned: 
  - Shipping & Receiving Team
  - All Employees
  - VPN Remote Workers
        ↓
System queries: What permissions do these groups have?
        ↓
Permissions determined:
  From "Shipping & Receiving Team":
    ✓ Create shipping labels
    ✓ Track shipments
    ✓ View monthly reports
    ✓ Access customer contact data
  
  From "All Employees":
    ✓ Access company intranet
    ✓ View company policies
    
  From "VPN Remote Workers":
    ✓ Connect to company VPN
    ✓ Access internal file server
        ↓
Combined permissions: [All of above]
        ↓
User can perform these actions on resources they access
```

### Scaling Benefits

**Scenario Change: New Permission Needed**

❌ **Without Groups:**
```
All 1000 shipping employees need "Export shipment data" permission
Must update: 1000 individual user records
Time: 2-3 hours
Error risk: High (might miss users)
Verification: Very tedious
```

✅ **With Groups:**
```
Shipping & Receiving Group needs "Export shipment data" permission
Update: 1 group record
Add permission: "Export shipment data" to group
Time: 5 minutes
Error risk: Low (one change)
Affected users: All 1000 automatically (1000 ÷ 1 = just 1 change!)
Verification: Simple (verify one group)
```

**Scaling Mathematics:**

```
Direct model:      N users × M permissions = N×M mappings
Group model:       N users ÷ G groups = N/G group memberships
                   G groups × M permissions = G×M permission mappings
                   Total: (N/G) + (G×M) vs (N×M)

Example with 1000 users, 10 permissions, 10 groups:
Direct:    1000 × 10 = 10,000 mappings
Groups:    (1000/10) + (10×10) = 100 + 100 = 200 mappings
           Reduction: 98%!
```

## Best Practices

### Authentication Best Practices

1. **Use Strong Authentication**
   - Passwords: Minimum 12 characters, complexity requirements
   - Multi-factor: Require MFA for sensitive systems
   - Certificates: Use for device authentication

2. **Centralize Authentication**
   - Use centralized AAA servers (RADIUS, TACACS+, LDAP)
   - Consistent policies across all systems
   - Easier to manage and audit

3. **Implement Strong Password Policies**
   - Minimum length (12+ characters)
   - Complexity requirements (upper, lower, numbers, symbols)
   - Password history (prevent reuse)
   - Expiration policy (change regularly)
   - Account lockout (prevent brute force)

4. **Multi-Factor Authentication**
   - Require for remote access
   - Require for privileged accounts
   - Use non-SMS factors where possible (hardware tokens, TOTP)

5. **Account Management**
   - Regular access reviews
   - Timely provisioning for new users
   - Timely deprovisioning for departing users
   - Disable inactive accounts

### Authorization Best Practices

1. **Implement Principle of Least Privilege**
   - Users get minimum access needed for their job
   - Not all employees should be admins
   - Regular review of excessive permissions

2. **Use Appropriate Authorization Model**
   - Small org: Role-Based (RBAC)
   - Complex environment: Attribute-Based (ABAC)
   - Traditional environment: Organization-Based

3. **Use Group-Based Management**
   - Create groups matching job functions
   - Assign users to groups (not individual permissions)
   - Assign permissions to groups
   - Simplifies scaling

4. **Regular Access Reviews**
   - Quarterly review of user permissions
   - Verify users still need their access
   - Remove unnecessary permissions
   - Document justifications

5. **Separate Duties**
   - No single person controls entire critical process
   - Requires multiple approvals for sensitive actions
   - Prevents fraud and errors
   - Example: Finance - requester, approver, and payment maker

6. **Document Authorization Policies**
   - Clear policy on what each role can access
   - Different policies for different sensitivity levels
   - Review and update regularly

### Accounting Best Practices

1. **Enable Comprehensive Logging**
   - Log all authentication attempts (success and failure)
   - Log all authorization decisions
   - Log all resource access (files, data, systems)
   - Log all administrative actions

2. **Secure Log Storage**
   - Store logs on secure, separate systems
   - Protect against tampering and deletion
   - Use write-once storage where possible
   - Encrypt logs at rest and in transit

3. **Regular Log Review**
   - Monitor for suspicious patterns
   - Alert on multiple failed login attempts
   - Alert on unusual access patterns
   - Investigate alerts promptly

4. **Maintain Appropriate Retention**
   - Keep logs long enough for forensics (usually 1+ years)
   - Meet regulatory requirements
   - Archive old logs separately
   - Securely delete expired logs

5. **Log Aggregation and Analysis**
   - Centralize logs from all systems
   - Use SIEM (Security Information and Event Management)
   - Correlate logs from multiple sources
   - Automated alerting for suspicious activity

6. **Create Audit Trails**
   - Chain of custody for sensitive data
   - Document who accessed what and when
   - Support for forensic investigations
   - Compliance with regulations

## Implementation Considerations

### Architecture Decisions

**Centralized vs. Distributed Authentication**

| Aspect | Centralized | Distributed |
|--------|---|---|
| **Architecture** | One AAA server | Multiple servers |
| **Scalability** | Single bottleneck | Scalable |
| **Consistency** | Single source of truth | Possible inconsistency |
| **Resilience** | Single point of failure | More resilient |
| **Management** | Simple | Complex |

**Recommendation:** Centralized with high availability and failover

### Technology Choices

**AAA Protocols:**
- **RADIUS** - Remote Authentication Dial-In User Service (legacy)
- **TACACS+** - Terminal Access Controller Access-Control System Plus (more secure)
- **LDAP** - Lightweight Directory Access Protocol (directory-based)
- **OAuth/OIDC** - Modern web-based authentication
- **Kerberos** - Enterprise single sign-on

**Directory Services:**
- **Active Directory** - Microsoft enterprise standard
- **LDAP** - Open-source alternative
- **FreeIPA** - Red Hat identity management
- **Keycloak** - Open-source identity provider

## Key Takeaways

1. **AAA is foundational** - Every access decision requires all three: authentication, authorization, accounting

2. **Identify, Authenticate, Authorize** - First claim identity, then prove it, then determine what you can access

3. **Centralized AAA servers** - Enable scalability and consistency across all systems

4. **Device authentication** - Use certificates for computers and devices

5. **Use authorization models** - Don't use direct user-permission assignments; use groups and roles for scalability

6. **Group-based management scales** - Dramatically reduces complexity as organization grows

7. **Accounting for compliance** - Logging and tracking enable forensics, compliance, and security monitoring

8. **Principle of least privilege** - Users should have minimum access needed for their job

9. **Regular reviews essential** - Access rights change over time; regular reviews catch orphaned permissions

10. **Consistent policies** - Implement same AAA controls across all systems for consistency and security
